# Bazel Vector SIP Module

A Bazel repository rule for integrating Software Integration Packages (SIP) into the Bazel build.

## Overview

This module provides the `bsw` repository rule that automates the download (or local path usage) and "bazelization" of Vector MICROSAR SIP.

## Features

- 🌐 **Download from URL** or 📁 **Use local file path**
- 🔐 **Authenticated downloads** via `.netrc` (Artifactory, Nexus, etc.)
- 🤖 **Automatic component discovery** from directory structures
- 📦 **Unity build support** for SIP components

## Requirements

- **Bazel 7.0+** (uses Bzlmod)
- **Network access** (only if downloading from URL)

## Installation

Add to your `MODULE.bazel`:

```python
bazel_dep(name = "rules_bsw", version = "0.0.1")

# For local development
local_path_override(
    module_name = "rules_bsw",
    path = "../rules_bsw",
)
```

## Usage

### Option 1: Download from URL

```python
load("@rules_bsw//:defs.bzl", "bsw")

bsw(
    name = "bsw",
    url = "https://artifactory.example.com/sip-v2.5.0.zip",
    integrity = "sha256-abc123...",
    strip_prefix = "SIP-2.5.0",  # Optional strip_prefix
    auth_patterns = {
        "artifactory.example.com": "netrc",
    },
)
```

### Option 2: Use Local Directory

```python
load("@rules_bsw//:defs.bzl", "bsw")

bsw(
    name = "bsw",
    path = "D:/Projects/MSRC/SIP-2.5.0",
)
```

### Option 3: Use Local Archive

```python
load("@rules_bsw//:defs.bzl", "bsw")

bsw(
    name = "bsw",
    path = "D:/Downloads/sip-v2.5.0.zip",
    strip_prefix = "SIP-2.5.0",  # Optional strip_prefix for archives
)
```

## Generated Structure

After bazelization, the SIP will have:

```
@bsw/
├── BUILD.bazel                          # Root with DaVinci exports
└── Components/
    ├── Can_30_Core/BUILD.bazel          # Can_30_Core_headers, Can_30_Core_sources
    ├── CanIf/BUILD.bazel
    ├── Com/BUILD.bazel
    └── ...
```

## Important Limitations

### Scope of This Rule

This rule **only makes the static parts of the SIP source files available** for Bazel. It does not provide finished, ready-to-use build targets.

**What this rule does:**
- ✅ Discovers and exposes BSW component headers and sources
- ✅ Creates `_headers` and `_sources` targets for each component
- ✅ Handles Unity build detection

**What this rule does NOT do:**
- ❌ Create complete, buildable component libraries
- ❌ Generate configuration data (gendata)
- ❌ Provide callout implementations
- ❌ Set up component dependencies automatically

**You must** define your own `cc_library` targets that combine:
1. BSW sources (from this rule)
2. Generated configuration files (from configuration generation tools)
3. Callout implementations (user code)
4. Component inter-dependencies

### Third-Party MCAL Support

**Third-party MCAL drivers are NOT supported** by this rule. The `bsw` repository rule is designed exclusively for Vector MICROSAR SIP packages.

If you need to integrate third-party MCAL drivers, you must:
- Create separate Bazel targets for those drivers
- Manage their dependencies independently
- Ensure compatibility with your BSW component libraries

## Defining the Build Targets

### Understanding Generated Targets

Each BSW component generates two targets:
- `<Component>_headers` - A `cc_library` with `hdrs` and `includes` set to `["Implementation"]`
- `<Component>_sources` - A `filegroup` containing `.c` files (or Unity `.c` files)

**Important:** Since the `_headers` target is a `cc_library` with `includes` already configured, you don't need to specify include paths via `copts`. The headers will be available automatically.

### Component Library Structure

A complete component library typically consists of three parts:
1. **Headers and sources** from the BSW package (generated by this rule)
2. **Generated data** (`gendata`) from component configuration. See generation rules for more information
3. **Callout files** (user-provided implementations)

### Specifying Dependencies

When creating component libraries:

**✅ Correct:** Reference headers in `deps` (or `implementation_deps`):
```python
cc_library(
    name = "Can_30_Core",
    srcs = [
        "@bsw//Components/Can_30_Core:Can_30_Core_sources",
        # Add gendata and callouts here
    ],
    implementation_deps = [
        "@bsw//Components/Can_30_Core:Can_30_Core_headers",
        "@bsw//Components/Det:Det_headers",  # Component dependencies
        "@bsw//Components/CanIf:CanIf_headers",
    ],
)
```

**❌ Incorrect:** Do NOT use `hdrs` attribute for BSW component headers:
```python
cc_library(
    name = "Can_30_Core",
    hdrs = [
        "@bsw//Components/Can_30_Core:Can_30_Core_headers",  # Wrong!
    ],
)
```

### Component Dependencies

Each BSW component may depend on headers from other components. You must specify these dependencies explicitly in your component library targets:

```python
cc_library(
    name = "CanIf",
    srcs = [
        "@bsw//Components/CanIf:CanIf_sources",
        "//gendata:CanIf_Cfg",
        "//callouts:CanIf_Callouts.c",
    ],
    implementation_deps = [
        "@bsw//Components/CanIf:CanIf_headers",
        # Add dependencies to other component headers
        "@bsw//Components/Can_30_Core:Can_30_Core_headers",
        "@bsw//Components/Det:Det_headers",
        "@bsw//Components/EcuM:EcuM_headers",
    ],
)
```

### Example: Complete BSW Integration

```python
# Complete component library with all three parts
cc_library(
    name = "BswM",
    srcs = [
        "@bsw//Components/BswM:BswM_sources",         # BSW sources
        "//appl:gendata_BswM",                        # Generated configuration
        "//source/callouts:BswM_Callout_Stubs.c",     # User callouts
    ],
    implementation_deps = [
        "@bsw//Components/BswM:BswM_headers",         # Component headers
        "@bsw//Components/PduR:PduR_headers",         # Dependencies
        "@bsw//Components/Det:Det_headers",
        "@bsw//Components/Com:Com",
        "//appl:gendata_Com",                         # Generated headers
    ],
)
```

## Advanced Configuration

### Authentication for URL Downloads

Create `~/.netrc` for authenticated downloads:

```
machine artifactory.example.com
login your-username
password your-token-or-password
```

Or set `NETRC` environment variable:
```powershell
$env:NETRC = "C:\path\to\.netrc"
bazel build //...
```

### Unity Builds

If a SIP component contains `*_Unity.c` files, the bazelization script:
1. Treats regular `.c` files as headers (they're `#include`d by the Unity file)
2. Only compiles the Unity file

This reduces compilation units and speeds up builds.

### DaVinci Configurator

#### cfg5
If a DaVinci Configurator 5 (cfg5) is present in the BSW package, it will be automatically made available for Bazel builds. The configurator executables are exported from the root BUILD.bazel:
- **Windows**: `DVCfgCmd.exe`, `DaVinciCfg.exe` / `DaVinciCFG.exe`
- **Linux**: `DVCfgCmd` (if Linux-specific cfg5 is provided)

**Linux cfg5 Replacement:**
If you're on Linux and the BSW package contains a Windows-only cfg5, you can specify a Linux version separately:

```python
bsw(
    name = "bsw",
    url = "https://artifactory.example.com/sip-v2.5.0.zip",
    integrity = "sha256-abc123...",
    cfg5_linux_url = "https://artifactory.example.com/cfg5-linux.tar.gz",
    cfg5_linux_integrity = "sha256-def456...",
    cfg5_linux_strip_prefix = "DaVinciConfigurator",
    auth_patterns = {"artifactory.example.com": "netrc"},
)
```


#### R36+ (cfg6)
For R36 and later releases, the new DaVinci Configurator 6 (cfg6) is delivered **independently** from the BSW package and must be consumed as a separate dependency within the repository.

## Repository Rule Reference

### `bsw`

| Attribute | Type | Required | Description |
|-----------|------|----------|-------------|
| `url` | `string` | One of `url`/`path` | URL of SIP ZIP package |
| `path` | `string` | One of `url`/`path` | Local file system path (directory or archive) |
| `integrity` | `string` | No | Integrity hash (e.g., `sha256-...`) (only for `url`) |
| `strip_prefix` | `string` | No | Strip prefix for extracted archive |
| `netrc` | `string` | No | Path to `.netrc` file (only for `url`) |
| `auth_patterns` | `string_dict` | No | URL pattern → auth method mapping |
| `cfg5_linux_url` | `string` | No | URL for Linux cfg5 tool (replaces DaVinciConfigurator on Linux) |
| `cfg5_linux_integrity` | `string` | No | Integrity hash for Linux cfg5 tool |
| `cfg5_linux_strip_prefix` | `string` | No | Strip prefix for Linux cfg5 tool archive |

**Note:** You must specify **exactly one** of `url` or `path`.

## Local Path Behavior

### Directory Path
When `path` points to a directory:
- Contents are **symlinked** into the repository
- If `strip_prefix` is specified, only `path/strip_prefix/*` is symlinked
- Changes to the source directory are **not** automatically detected (run `bazel clean` to refresh)

### Archive Path
When `path` points to an archive (`.zip`, `.tar.gz`, etc.):
- Archive is extracted into the repository
- `strip_prefix` strips leading directory components (like URL mode)
- Supported formats: `.zip`, `.tar.gz`, `.tgz`, `.tar.bz2`, `.tar.xz`

## Architecture

```
rules_bsw/
├── defs.bzl                  # Public API exports
├── BUILD.bazel               # Module root
├── MODULE.bazel              # Bzlmod metadata
└── private/
    └── bsw.bzl      # Repository rule implementation
```

**Execution flow:**

1. **Repository fetch phase**: Bazel downloads archive or symlinks local path
2. **Execution**: Create BUILD files for components
3. **Build phase**: Generated targets become available as `@bsw//Components/...`

## Troubleshooting

**Local path does not exist:**
```
Error: Local SIP path does not exist: D:/path/to/sip
```
→ Verify the path is correct (use absolute paths)

**Both url and path specified:**
```
Error: Cannot specify both 'url' and 'path'
```
→ Choose only one source for the SIP package

**Prefix not found in local path:**
```
Error: Prefix 'SIP-2.5.0' not found in local path: D:/sip
```
→ Verify the `strip_prefix` matches the directory structure

**Components not discovered:**
```
Discovered 0 components
```
→ Check that `Components/` directory exists with `*/Implementation/` subdirectories
